# Follow-up


## 1. 场景应对

### 1.1 如果这套系统后续由你负责

原型已完成 GB/T 1568-2008 的 OCR → 分块 → BM25 检索 → DeepSeek 生成 → 自检全链路，可在终端交互问答。因为目前仅是少量问题抽样测试，覆盖度不够，下一步需要进行验证闭环，量化检索命中率与答案准确率，然后需要将检索命中率和答案可用率进一步确认和提高。

### 1.2 如果要给客户试用

**交付前需要与产品、客户、技术负责人确认的事项：**

**与产品确认：**
- 使用场景与用户画像：最终用户是谁（质检工程师、标准化人员、业务开发），使用频率和并发量预期
- 文档范围：第一期支持几份标准文档，文档类型（国标、企标、行标），PDF 质量（扫描件/电子版/混合）
- 功能边界：定义系统哪些能做和不能做，交互方式的范围，输入输出的格式
- 验收标准：具体的可量化指标（检索命中率 > 85%，答案可用率 > 90%），"可用"的定义需要共识

**与客户确认：**
- 核心痛点：客户现在怎么查标准，最关心的"第一分钟体验"是什么，错误答案容忍度
- 部署环境与合规：数据能不能出内网（决定用云 API 还是本地模型），PDF 是否可外传做 OCR，是否有 GPU 资源
- 试用计划：试用用户范围、周期、反馈收集机制、试用期"叫停"标准
- 访问控制：是否需要登录/权限控制，不同角色是否看到不同文档范围

**与技术负责人确认：**
- OCR 方案：目标 Linux 环境用 PaddleOCR 还是 Tesseract，是否需要对比评估
- 检索方案：是否从 BM25 迁移到 ChromaDB 向量检索
- 模型方案：DeepSeek API 的 token 成本，是否需要本地模型 fallback
- 部署与运维：服务器配置、Docker 环境、日志监控、API 降级策略
- 数据与安全：API Key 管理方式、问答日志脱敏与保留策略、审计日志需求
- 集成方式：独立 Web 应用还是提供 API，接口格式、鉴权方式、速率限制

**第一版交付范围：** 围绕客户核心痛点抽象出核心功能，完成客户核心功能并达到可演示的状态。建议裁剪为 3-5 份同类型国标、Web 问答 + `/ask` API、Docker 单机部署。

### 1.3 如果要保证质量和稳定性

检索命中率 > 85% 和答案准确率 > 90%，才能基本确认系统可靠。

如果演示或生产前出现问题：
- 及时向客户做出客观说明，明确演示版本可能存在未知问题，但保证最终上线版本稳定可靠
- 快速定位：确认问题和报错日志，反馈给 AI 工具进行排查
- 止损：评估影响范围，决定回滚、hotfix 还是降级方案
- 修复：修复后跑全量测试集验证，确保不引入回归
- 确认修复有效：同一 bad case 复现 → 修复 → 验证通过，形成闭环

### 1.4 如果时间有限

- **重要紧急优先处理**，重要不紧急进行必要取舍
- 优先安排核心功能的实现和验证，这个系统最核心的是 OCR 和检索
- PDF 识别中表格识别等攻坚问题，需要抛出风险，向产品、技术相关同事同步，申请支持
- 非核心功能可以协调其他资源协同完成

### 1.5 如何使用 AI 或智能体工具

- **编程**：要求 AI 必须先输出方案，再进行代码实现（plan 模式），代码提交前必须 review 并进行必要验证
- **排查问题**：将报错日志和上下文提供给 AI 辅助定位
- **快速学习**：多领域知识查阅、文档汇总、热点收集等
- **必须自己把关的**：代码 review、问题排查结论核对、测试 case review、文档校对

### 1.6 补充一个真实例子

最早负责广告投放平台系统时，同时负责投放素材管理和任务中心管理两个系统，开发周期只有两周。素材管理涉及第三方媒体平台联调，有复杂的复制逻辑和异步任务状态计算逻辑。及时将问题跟产品沟通，明确了重要性和影响深度，产品同意对部分非核心功能进行取舍，拆分为 2.0 版本后期二次迭代开发。开发过程中采用每日站立会同步策略，及时暴露问题并解决问题，防止卡点，最终保证了项目如期上线和交付质量。

### 1.7 如果文档章节层级解析不对

假设系统解析后发现不少文档的章节层级不准确（标题、条款、段落、表格归属、跨页内容关系识别错），导致后续检索和问答效果受影响。

**定位问题：** 首先对解析有问题的文档进行问题分类（OCR 噪声、表格归属、跨页断裂、层级错误），根据不同问题分而治之。

**判断影响范围：** 用测试集跑分，标出受影响的题目，统计占比作为修复优先级依据。

**短期方案：**
- 将整个表格作为 chunk 保留，不拆分
- 增加跨页合并逻辑
- 条款编号白名单校验

**长期方案：**
- 利用 PyMuPDF 的文本块信息（位置坐标、字号、缩进）重建章节层级
- 使用大模型的 vision 能力，PDF 转图后进行版面识别

**验证修复有效：** 修复前后同一套测试集跑分对比，针对性用例逐个验证，人工抽检知识库质量。

### 1.8 如果系统效果不稳定

**当前最核心问题：** PDF 中表格内容的识别，直接影响到检索命中率，检索命中率和答案准确率都与这个问题强相关。

**处理路径：**
1. 广泛收集产品验收与用户使用过程中遇到的问题，定义评价标准
2. 针对性排查和定位 bad case
3. 尝试优化方案（强化表格内容识别、采用 AI vision 能力等）
4. 重新针对表格相关问题进行验证，确认优化方案有效
5. 判断优化是否真的有效：跑全量测试集 + 人工抽检，而非凭感觉

## 2. 工作方式补充

### 2.1 遇到不清楚但又比较急的需求

满足用户需求是所有工作的起点。遇到这种情况，优先与产品、客户沟通，了解深层需求以及紧要程度，思考是否可以小成本快速给出反馈。如果不能，考虑向上反馈寻求帮助。如果讨论后依然没有快速实现方案，需要给对方明确说明，争取空间和时间进行需求拆解和交付。总之，必须有反馈，要让对方感受到你重视他的需求。

### 2.2 关键节点前发现问题

如果周五晚上发现系统还有一个影响演示或交付的问题，但周一就要给客户看：

- 如果与核心流程无关，可以在演示中忽略，同步产研同事情况
- 如果是核心流程，需要快速处理，同步产研负责人，协调必要资源进行快速修复，力保周一演示流畅

### 2.3 沟通中出现分歧

如果产品、客户或技术同事对同一个问题的理解不一致，甚至有人比较着急：

- 最好的办法是快速结束当前讨论，单独与有分歧人员沟通，了解分歧的根本原因
- 所有方案的定调始终要以客观事实为准
- 完成沟通后，需要最终达成共识
- 大多数人的意见也未必正确，相信科学，但也要尊重人性，任何问题都有解法

### 2.4 多件事情同时压过来

如果同一时间有开发、问题排查、测试验证、文档说明几件事都要推进：

- 根据任务优先级和重要程度稳步安排
- 这些工作都可以依赖 AI 来提高工作效率
- **必须自己复查的**：代码必须 review、问题排查结论必须仔细核对、测试 case 也要 review、文档说明需要校对
