# Follow-up

## 1. 推进计划

### 1.1 当前状态
原型已完成 GB/T 1568-2008 的 OCR → 分块 → BM25 检索 → DeepSeek 生成 → 自检全链路，可在终端交互问答。

### 1.2 下一步里程碑

| 阶段 | 目标 | 预计周期 |
|------|------|----------|
| 验证闭环 | 构造 30 个标准问答对，量化检索命中率与答案准确率 | 1 周 |
| 质量达标 | 检索命中率 > 85%，答案可用率 > 90%（人工抽检） | 2 周 |
| MVP 交付 | Web 界面 + 多文档支持，可在团队内部试用 | 3 周 |
| 正式交付 | 接入内部标准库，提供 API 服务 | 4 周 |

### 1.3 关键动作
- **每周评审**：每周五输出一页进展简报（评估指标变化、bad case 分析、阻塞项）
- **Bad case 驱动迭代**：从验证中收集的 bad case 反推检索/生成策略调整，而非猜测式优化
- **先跑通再扩展**：在 GB/T 1568 单文档上把准确率做到可交付水平，再扩展其他标准

## 2. 验证方法

### 2.1 测试集构建
从 GB/T 1568-2008 中人工标注至少 30 个问答对，覆盖以下维度：

| 类别 | 示例 | 预期占比 |
|------|------|----------|
| 数值提取 | "键的抗拉强度要求是多少？" | 30% |
| 表格查询 | "45 钢的硬度要求？" | 20% |
| 定义解释 | "什么是钩头楔键？" | 20% |
| 跨条款综合 | "键的尺寸公差和形位公差各有哪些要求？" | 15% |
| 边界/拒答 | "这个标准适用螺丝吗？" | 15% |

### 2.2 评估指标

每次修改检索或生成逻辑后，跑全量测试集：

- **检索命中率（Recall@5）**：正确答案所在条款是否在 Top-5 检索结果中
- **答案准确率（人工）**：答案中关键事实（数值、牌号、条款号）是否与原文一致
- **拒答准确率**：对标准范围外的问题是否正确拒答
- **幻觉率**：自检发现或人工发现的编造内容占比

### 2.3 评估脚本
在 `pdf_qa_agent.py` 旁新增 `eval.py`，输入为 JSON 格式的测试集，输出上述指标报告，每次改动后一键跑分。

## 3. 交付方式

### 3.1 MVP 交付物
- **Web 界面**：Gradio 单页应用，上传 PDF → 自动构建知识库 → 问答
- **API 端点**：`POST /ask` 接受 question 返回 answer + sources，可被其他系统集成
- **Docker 镜像**：包含所有依赖（剔除 macOS Vision OCR），使用 PaddleOCR 或 Tesseract 替代

### 3.2 交付 Checklist
- [ ] 测试集覆盖率和指标达标
- [ ] 至少 3 份内部标准文档实测通过
- [ ] README 更新为交付版（部署步骤、API 文档、示例）
- [ ] 异常处理完善（PDF 损坏、OCR 失败、API 超时等边界情况）
- [ ] 确定运维负责人和反馈渠道

### 3.3 目标交付环境
- 公司内网 Linux 服务器部署
- DeepSeek API 或内部部署的开源模型（如 Qwen）均可切换
- ChromaDB 持久化向量库替代 JSON 文件
